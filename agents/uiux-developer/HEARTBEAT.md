# UI/UX Developer Heartbeat

## Schedule

**Frequency**: Weekly (Monday afternoon — after product-owner sets priorities)

## Each Cycle

### 1. Read Context
- Read `knowledge/AUDIENCE.md` for user personas and pain points
- Read `knowledge/BRAND.md` for visual guidelines
- Read `knowledge/STRATEGY.md` for current priorities
- Read recent `journal/entries/` for requirements from product-owner, user feedback
- Read own `MEMORY.md` for proven design patterns

### 2. Assess State
- Are there new features needing design?
- Is there user feedback to analyze?
- Are there design inconsistencies to fix?
- Did frontend-developer report UX issues?

### 3. Execute Skill (Decision Tree)
- New user feedback or feature area → **USER_RESEARCH** skill
- Feature needs initial design → **WIREFRAMING** skill
- Feature needs interaction details → **INTERACTION_DESIGN** skill

### 4. Log to Journal
- Designs created
- Research findings
- Interaction patterns defined
- Handoff notes for frontend-developer
- Next steps

## Weekly Review

1. **Gather Data**: UX issue reports, design handoff feedback from frontend
2. **Score Against Targets**: Task completion rate, design consistency, rework rate
3. **Analyze Wins and Misses**: Which designs shipped smoothly? Which needed rework?
4. **Update Memory**: Add proven design patterns, user insights
5. **Log Weekly Summary**: Performance vs targets, user signals

## Escalation Rules

- Requirement too vague to design → handoff to product-owner
- Design conflicts with technical constraints → collaborate with backend/frontend developers
- User research reveals strategic pivot needed → escalate to human
- Design system needs new component category → propose and document
